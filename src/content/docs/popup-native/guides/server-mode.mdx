# Server Mode

`Mode.Server` is the repository's primary integration path. Both the Android demo and the iOS demo initialize the SDK in Server Mode.

## How it works

When initialized in Server Mode, the SDK:

1. reads `popupOptions.publicKey`
2. fetches popup definitions from the Deepdots backend
3. normalizes remote payloads into local popup definitions
4. evaluates segmentation, cooldowns, and triggers inside the client
5. renders the selected popup and survey flow natively

## Remote endpoints used by the SDK

The default popup service derives the backend base URL from `InitOptions.environment`:

- `Environment.Production` (default): `https://api.deepdots.com`
- `Environment.Development`: `https://api-dev.deepdots.com`

`InitOptions.debug` is independent of the environment and only controls SDK log output.
To talk to production while keeping logs on, set `debug = true` and leave `environment`
at its default (or set it explicitly to `Environment.Production`).

The SDK fetches popups from:

```text
/sdk/{publicKey}/popups
```

The SDK posts popup lifecycle events to:

```text
/sdk/popups
```

## Trigger types supported in code

- `TimeOnPage(seconds)`
- `Scroll(percentage)`
- `Exit(delaySeconds)`
- `Event(name)`
- `Click(targetId)`

## Segmentation supported in code

Remote popup definitions can include:

- `lang`
- `path`

The host app is responsible for supplying both:

- language through `provideLang`
- current screen or route through `setPath(...)`

## Cooldown handling

Frequency caps and re-display rules are enforced by the backend on every `/sdk/{publicKey}/popups` fetch, so they survive app restarts without any host-side configuration.

Cooldown evaluation supports:

- `SHOWED`
- `PARTIAL`
- `COMPLETED`

## Best fit

Server Mode is the right choice when teams want:

- central control over popup definitions
- reduced app release dependency for content or targeting changes
- segmentation and trigger logic distributed through backend-managed popup payloads
